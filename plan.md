Below is a structured QA review of the **Paksa Financial System** repository.  I inspected both the front‑end and back‑end directories, focusing on router configuration, layout composition, navigation, state management and incomplete or inconsistent implementations.  Issues are grouped by severity (critical → low) and the likely real‑time impact on the application.


## 🔴  Critical Issues (High Impact)
| **Router/Layout configuration**        | **Nested layout rendering** – Many routes specify a layout in `meta.layout` **and** set the same layout as the route’s component. In `/auth` the route uses `component: () => import('@/layouts/AuthLayout.vue')` while also specifying `layout: 'AuthLayout'`. `App.vue` wraps each page in a layout based on `meta.layout`, so this causes a second `AuthLayout` (or `MainLayout`) to render inside the first.  The result is duplicated sidebars/headers and multiple versions of the login page (as seen in your screenshots).  Similarly, module routes use `component: MainLayout.vue` and `layout: 'MainLayout'`, but `MainLayout` isn’t registered in `App.vue`, so the router defaults to `AppLayout` and nests another layout inside, compounding duplication. | **Critical UX bug:** Confusing UI, wasted space, and navigation errors on all pages using those routes.                                                               |
| **Missing/mismatched components**      | Several routes in `router/modules/allRoutes.ts` refer to components that do not exist (e.g., `@/views/auth/Login.vue`, `@/views/auth/Register.vue`).  The actual auth views are under `src/modules/auth/views`.  Navigating to these routes will break.                                                                                        | **Critical runtime failure.** Users will encounter blank pages or 404s.

## 🟠  Mid‑level Issues (Medium Impact)
| **Feature completeness**  | Many front‑end modules contain placeholders and comments such as “`// TODO: Implement export functionality`” (`reportUtils.ts`), “`// TODO: Replace with actual API calls`” (journal entries), and “`// TODO: Call API to get trial balance data`”.  This suggests key features (report exports, save/import, data retrieval) aren’t finished. | Users will hit dead‑ends when trying to export reports, save entries, or fetch data, which diminishes trust in the system. |
| **Code duplication and .bak files** | Files like `Dashboard.vue.bak` and `JournalEntries.vue.bak` remain in the repo alongside their active versions.  These backups increase maintenance cost and risk that developers edit the wrong file.                                                                                          | Potential confusion for developers and testers; risk of shipping outdated code.                                            |
| **Layout registration**             | `MainLayout.vue` is used in route definitions but not registered in `App.vue`’s `layouts` map.  Thus the router falls back to `AppLayout`, causing inconsistent styling and nested components.                                                                                     | Users may see UI inconsistencies and unpredictable navigation because different pages fall back to the wrong layout.       |
| **Token storage**                   | The `auth.store.ts` saves authentication tokens directly to `localStorage`/`sessionStorage` and sets a dummy user.  There’s no token encryption or expiry handling.  If a user checks “remember me,” the token is persisted in plain text.                                                                                                     | Moderate security risk; tokens could be stolen from the browser, leading to unauthorized access.                           |
| **Inconsistent file paths**         | Multiple modules import files with capitalisation mismatches or wrong folders (e.g., `@/modules/accounts-payable/views/APDashboard.vue` vs. `VendorsView.vue` vs. `BillProcessingView.vue`), which could break on case‑sensitive file systems.                                                                                                 | Runtime errors on some operating systems and increased time spent debugging path issues.                                   |
| **Logging and error messages**      | Many services log exceptions to console and rethrow generic error messages.  This pattern can clutter server logs and doesn’t provide useful feedback to clients.                                                                                                                                                                              | Harder to diagnose issues in production; limited user feedback when errors occur.                                          |


## 🟢  Low‑severity Issues (Low Impact)

| **Styling overlap** – Several component `<style>` blocks are unscoped or rely on global CSS variables.  This can cause style leakage between components, especially when layouts are nested incorrectly. | Minor visual glitches; can usually be fixed with scoped CSS.             |
| **Unused code** – There are several placeholder pages (e.g., `UnderConstruction.vue`) and unused menu items in `AppLayout.vue`.  | Adds to code bloat and confusion, though they don’t break functionality. |
| **Console logging** – Frequent use of `console.error` and `console.log` in production code may leak internal details or clutter logs.   | Low risk; should be replaced with a proper logging mechanism.            |
| **Duplicate navigation entries** – The top navigation and sidebar sometimes display the same items (`Dashboard`, `Reports`, etc.), reducing clarity for users.  | Can be refined by consolidating menu definitions.                        |

## ✅ Recommendations

1. **Fix router and layout nesting**: Remove layout components from route definitions and rely solely on the `meta.layout` property.  Register all layouts (including `MainLayout`) in `App.vue`, and ensure routes don’t import a layout they also specify in meta.
2. **Remove or update `router/modules/allRoutes.ts`**: It references non‑existent files and duplicates logic from `router/index.ts`.
3. **Complete TODOs in back‑end services**: Implement approval workflows, AI predictions, payroll journal entries, notification triggers and permission checks.  Until these are done, disable or clearly mark related features in the UI.
4. **Implement robust authentication**: Use real API calls for login, store tokens securely (e.g., HTTP‑only cookies), and handle token refresh and expiry properly.
5. **Audit file paths and remove `.bak` files**: Clean up unused backups and ensure all imports reference the correct files.
6. **Apply scoped styles**: Use `<style scoped>` wherever possible and verify that CSS variables are defined.
7. **Enhance error handling**: Replace generic `try/catch` logs with meaningful error responses; avoid logging sensitive data.

By addressing the critical issues first—especially the router/layout misconfiguration and missing back‑end logic—you’ll improve the stability and user experience of the Paksa Financial System and set a solid foundation for finishing the remaining features.
